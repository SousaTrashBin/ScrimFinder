 %%%%%%%%%%%%%%%%%%%%%%%%%%%%%% Motivation %%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
 \section{Motivation}

 ScrimFinder was born from the need for a more structured and data-driven approach to competitive "scrims" (practice matches) in League of Legends. While the official game offers matchmaking, it lacks the depth required for teams and individual players to organize balanced practice sessions, track progress through custom-built ranking systems (MMR), and gain actionable insights through post-game ML-driven analysis. By leveraging a massive 78GB dataset of EUW matches, ScrimFinder provides a platform that not only matches players based on skill but also analyzes their performance to offer personalized coaching tips. Our goal was to build a truly cloud-native application that demonstrates high availability, scalability, and advanced data processing capabilities.

 \subsection{Match History - Bruno Faustino}

 \begin{itemize}
  \item Local store of simplified match information relevant to the user (winning team, patch, creation date and length, roles and champions of each player, number of kills, deaths, assists, healing, damage,..., per match, per team and per player), that can be inserted internally by the matchmaking service by providing the match's ID in Riot's API, the id of the queue and the MMR deltas of each player in that match;
  \item Match filtering based on any criteria the user defines from the simplified information, including obtaining all matches of a given queue, time, patch, player, all matches where the player had the most kills, played with one of the provided champions, all matches where a team had at most a certain amount of deaths, healing ...;
  \item Match sorting based on match queue, time, patch and team stats;
  \item Match retrieval with paging to avoid fetching enormous amounts of data all at once;
  \item Soft deletion of matches from the history;
  \item New match data forwarding to the data analysers for training.
 \end{itemize}

 \subsection{Detail Filling - Bruno Faustino}

 \begin{itemize}
  \item Direct contact with Riot's API to obtain complete match and player information;
  \item Automatic filtering of information to focus more on the user's needs;

  \item Ingestion of raw match data in batches with automatic deduplication and metadata storage;
  \item Extraction and versioning of structured feature vectors for different analytical models;
  \item Creation and management of curated, optimized training datasets with configurable filtering;
  \item Asynchronous execution of ML training jobs for draft, build, and performance models;
  \item Real-time monitoring of training progress and lifecycle management of active jobs;
  \item Maintenance of a versioned model registry with performance metrics.

 \subsection{Data Analysis - Rodrigo Neto}

 \begin{itemize}
  \item Win probability prediction for team compositions;
  \item Identification of team synergies, counter-matchups, and win conditions;
  \item Evaluation of item build effectiveness and possibly suggestion of alternatives;
  \item Automated benchmarking of player performance metrics against global/rank-tier averages;
  \item Global champion performance tracking including win, pick, and ban rate analytics.
 \end{itemize}

 \subsection{Authentication --- Rodrigo Neto}

\begin{itemize}
 \item Secure user registration with username normalization, email uniqueness,
 and password strength validation (minimum 8 characters, mixed case, digit,
 and special character);
 \item RS256 JWT token issuance with unique JWT ID (\ttt{jti}) claims for
 traceability and revocation;
 \item Token validation via HTTPBearer scheme with session cache lookup in
 BigQuery \ttt{access\_sessions} table;
 \item Refresh token rotation: each refresh invalidates the previous refresh
 token and issues a new pair, preventing replay attacks;
 \item Single-session enforcement: concurrent logins from the same user
 invalidate prior access tokens;
 \item Global logout (\ttt{logout-all}) revokes all refresh tokens and
 deletes all access sessions for a user;
 \item JWKS (JSON Web Key Set) endpoint for external service validation
 (Istio service mesh integration).
\end{itemize}

 %%%%%%%%%%%%%%%%%%%%%% Non-Functional Requirements %%%%%%%%%%%%%%%%%%%%%%%%
 \section{Non-Functional Requirements}

 In accordance with the project guidelines, we implemented several non-functional improvements to ensure the system is robust, scalable, and secure.

 \subsection{Microservices Architecture - Diogo Sousa}

 The core architecture focuses on high availability and loose coupling through message-driven patterns.

 \begin{itemize}
  \item \bff{API Gateway}: Utilization of \bff{Traefik} for client request routing, rate limiting and API version management due to its native integration with Docker and Kubernetes, and ease of configuration.
  \item \bff{Database per Service}: Each microservice (Matchmaking, Ranking, History) maintains its own isolated PostgreSQL database, ensuring data encapsulation and independent scaling. Critical databases utilize a \bff{Master-Slave replication model} to ensure high availability and read-scalability.
  \item \bff{Retries} - Everyone: Failed API requests are resent a configured amount of times before being officially reported as a failure to prevent occasional communication errors from degradating the system.
  \item \bff{Timeouts} - Everyone: API requests taking too long to provide an answer are stopped early to contribute to the \bff{fail-fast} pattern.
  \item \bff{Circuit Breaker} - Everyone: After a configured number of API request failures to a service, posterior request attempts to that service will immediately fail for some time instead of taking long to warn the answer of what will likely be another failure, contributing to the \bff{fail-fast} pattern.
  \item \bff{Cache Aside} (Performance, Availability) - Bruno Faustino: Utilization of a \bff{Redis} cache to store Riot's API reponses to reduce the number of external API calls.
  \item \bff{Message-Driven (Pub/Sub)}: Utilization of \bff{RabbitMQ} for asynchronous, decoupled communication (e.g., Matchmaking publishing game results that are consumed by History and Training services).
 \end{itemize}

 \subsection{Serverless \& FaaS Integration - Bruno Faustino}

 %%%%%%%%%%%%%%%%%%%%%%%%%%%%% Tuning %%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
 \section{Tuning}

 System performance is optimized through precise resource allocation and automatic scaling policies.

 \subsection{Resource Requests and Limits}

 Each service is configured with baseline requests and hard limits to ensure fair scheduling and prevent OOM (Out of Memory) events.
 \begin{itemize}
  \item \bff{Java Services}: Typically 400m-500m CPU and 512Mi Memory.
  \item \bff{Training Service}: High resource allocation (600m CPU, 2Gi Memory) to handle ML workloads.
  \item \bff{Analysis Service}: 500m CPU and 1Gi Memory for real-time inference.
 \end{itemize}

 \subsection{Scaling Policies}

 \begin{itemize}
  \item \bff{Horizontal Pod Autoscaler (HPA)}: Enabled for all services, scaling from 1 to 3 replicas based on a 70-80\% \bff{CPU} utilization threshold.
  \item \bff{Vertical Pod Autoscaler (VPA)}: Enabled for all services, automatically adjusting the \bff{memory} needs of each pod (restarting them if needed), transpassing Argo CD cluster sync. It will keep the ratio between memory requests and limits, which is 1.0.
 \end{itemize}

 \subsection{Some Notes about Resource Requests/Limits and Scaling Policies}

 \begin{itemize}
  \item \bff{CPU Requests}: Set for the HPA.
  \item \bff{CPU Limits}: Unset to prevent pods from becoming slower not due to actual CPU pressure but due to an uneccessary limit we would be imposing (CPU is compressible so the pod's CPU would be throttled instead of the pod being killed).
  \item \bff{Memory}: The requests are the same as the limits so that the pod falls on Kubernetes' \bff{Guaranteed QoS class} for memory, meaning the pod will be one of the last to be evicted during memory pressure, while the VPA will adjust both accordingly.
  \item \bff{HPA and VPA Together}: While the cluster has both, they each target a different resource type (HPA $\rightarrow$ CPU, VPA $\rightarrow$ Memory), so they will not conflict with each other.
 \end{itemize}

 \subsection{Health Probes}

 Comprehensive health monitoring is implemented using SmallRye Health:
 \begin{itemize}
  \item \bff{Readiness Probes}: Ensure traffic is only routed if the pod is answering to the health requests. Called more often than the liveness probe to stop sending traffic for some time before officially restarting the pod if it still hasn't answered.
  \item \bff{Liveness Probes}: Automatically restart pods that enter a deadlocked state.
  \item \bff{Startup Probes}: Configured with extended failure thresholds (up to 60 for Training service) to accommodate longer initialization times during cold starts.
 \end{itemize}

 \section{Corrections Implemented after Feedback from Testing}

 Following the \bff{Cloudy Day} testing phase and internal smoke tests, several critical improvements were implemented to enhance system reliability and client experience.

 \subsection{Improved Error Handling in Player Registration - Diogo Sousa}
 Initial feedback indicated that the \texttt{POST /api/v1/matchmaking/players} endpoint returned a \texttt{500 Internal Server Error} when a duplicate UUID was provided, whereas a duplicate Discord username correctly returned a \texttt{409 Conflict}. We updated the \texttt{PlayerServiceImpl} to perform an explicit check for existing IDs before persistence. This ensures that all conflict scenarios (ID or Username) return a clear \texttt{409} status with a specific error message, improving the API's predictability.

 \subsection{Standardized Exception Mapping - Diogo Sousa}
 We refined the \texttt{GlobalExceptionHandler} to move away from generic \texttt{REMOTE\_SERVICE\_ERROR} codes for client-side issues. HTTP status codes are now accurately reflected in the error response payload, using \texttt{BAD\_REQUEST} or \texttt{NOT\_FOUND} where appropriate. This directly addresses feedback regarding the ambiguity of error messages under high load and validation failures.

 \subsection{Documentation Consistency - Diogo Sousa}
 OpenAPI annotations were updated to explicitly state uniqueness constraints and provide example payloads for complex enums, reducing the "trial and error" approach previously required by external testers.

 \subsection{Clarified Match History Documentation Descriptions - Bruno Faustino}
 OpenAPI match history descriptions were updated to inform users that the matches only show up in the history after being added via the matchmaking service API and to clarify to users that the history filters must be provided in the body of the request and not as request query parameters, due to size limitations of the request endpoint.

 \subsection{Improved Guidance  and Default Definitions - Rodrigo Neto}
 The instructions and descriptions were insufficient to allow the intended use of certain endpoints, specially the ones whose default fields cause errors if unset. Added better descriptions and default handling.

 %%%%%%%%%%%%%%%%%%%%%%%%%%% Conclusions %%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
 \section{Conclusions}

 ScrimFinder is a cloud-native platform for competitive League of Legends practice matches, built as a multi-language microservices architecture integrating Java/Quarkus services for real-time matchmaking, ranking, and match history with Python/FastAPI services for ML-driven predictive analysis and authentication. The platform processes a 78GB dataset of EUW matches to provide skill-based matchmaking, post-game analytics, and personalized performance insights.

 Functionally, the system covers five core domains. The Matchmaking Service manages queue lifecycle, lobby formation, and match state coordination with support for both standard and role-based queues. The Ranking Service handles ELO/MMR calculation, leaderboard management, and Riot account integration. The Match History Service provides a centralized repository with advanced filtering, sorting, and paging capabilities. The Training and Analysis Services form a complete ML pipeline --- from raw match ingestion and feature extraction through dataset curation, model training, and real-time inference for draft win-probability, build scoring, and player benchmarking. The JWT Manager secures the entire platform with RS256 token issuance, session management, and Istio-compatible JWKS endpoint distribution.

 Non-functionally, the architecture implements several cloud-native resilience patterns. Retries, timeouts, and circuit breakers protect inter-service communication from cascading failures. Redis caching minimizes external API calls to Riot's services. RabbitMQ enables asynchronous, decoupled event propagation between Matchmaking, History, and Training services. The Detail Filling Service leverages serverless Google Cloud Functions for cost-effective bursty API polling. Infrastructure is provisioned via Terraform, deployed through Helm, and synchronized via Argo CD GitOps workflows.

 Resource tuning reflects deliberate trade-offs between performance and stability. Java services run with 400--500m CPU requests and 512Mi memory, while Python ML services receive higher allocations (600m CPU, 2Gi memory for Training; 500m CPU, 1Gi memory for Analysis) to accommodate model loading and inference workloads. CPU limits are intentionally unset to prevent throttling under legitimate load, while memory requests equal limits to guarantee Kubernetes QoS class. HPA scales replicas from 1 to 3 based on 70--80\% CPU utilization, and VPA adjusts memory requirements independently without conflicting. Health probes --- readiness, liveness, and startup --- ensure traffic is only routed to healthy pods, with extended startup thresholds for ML services with longer cold-start times.

 The estimated monthly operational cost of \euro357 on GKE Spot instances, self-hosted PostgreSQL/Redis/RabbitMQ, and BigQuery storage represents a pragmatic balance between capability and affordability for a student project.

 Beyond the technical deliverable, this project was fundamentally an exercise in learning cloud-native architecture through direct experience. Implementing patterns like circuit breakers and health probes in response to actual CI failures and production timeouts, rather than as theoretical exercises, provided a concrete understanding of why these patterns matter. Working across language boundaries with gRPC contracts, managing infrastructure as code with Terraform, and debugging distributed system behavior in ephemeral GKE clusters were all challenges that no classroom assignment could fully replicate. The fact that we went through building, breaking, and rebuilding a system of this complexity,completely from scratch, making use of many tools we did not dream existed, as a team of three students was a different and very demanding  episode of our academic journey, that despite the hardships allowed us to overcome troubles and ultimately get a lot of experience that we would have never gotten otherwise. We hope,this adventure will prove to be valuable in the future (but we hope it does not repeat itself).
